Backend   Mission   and   Architecture   Philosophy     1.1   What   the   Backend   Must   Achieve   The   Roomzy   backend   is   the   central   intelligence   of   the   entire   platform.   It   enforces   every   business   rule,   manages   every   state   transition,   controls   every   permission,   and   records   every   critical   action.   The   frontend   is   the   face   —   the   backend   is   the   brain.     Every   feature   visible   to   a   student,   hostel   owner,   or   admin   is   powered   by   a   secure,   modular   NestJS   API.   The   backend   must   guarantee   three   things   above   all   else:   trust   (only   verified   properties   are   visible),   safety   (only   authorized   users   can   perform   actions),   and   accuracy   (every   booking   state   and   payment   record   is   correct).     1.2   Platform   Mission   Roomzy   solves   the   critical   problem   of   students   struggling   to   find   safe,   verified,   and   affordable   housing   near   institutes.   The   backend   powers   a   physically-verified,   commission-based   marketplace   —   going   beyond   a   simple   listing   site   to   become   a   complete   student   living   ecosystem.    Every   hostel   is   physically   inspected   before   going   live.   Every   review   is   from   a   verified   stay.   Every   booking   payment   flows   through   the   platform.   These   guarantees   are   enforced   entirely   at   the   backend   layer .     1.3   What   Makes   Roomzy   Different   —   Backend   Perspective   Differentiator   Backend   Implementation   Physical   Verification   Hostel   status   state   machine   with   6   states. This   is   strategically   controlled   to   prevent   deal   bypass   —   students   cannot   simply   get   the   owner's   contact   details   and   take   the   transaction   outside   Roomzy.     •   Maximum   5   messages   allowed   before   booking   request   is   required   to   continue   the   conversation.   •   Phone   number   regex:   any   Pakistani   phone   format   (03XX-XXXXXXX,   +923XXXXXXXXX)   is   automatically   replaced   with   [Phone   number   hidden   —   submit   a   booking   request   to   receive   contact   details].   •   External   links   detected   and   blocked:   messages   containing   http://   or   https://   show   an   error:   'External   links   are   not   allowed   in   pre-booking   chat.'   •   Owner   contact   info   (phone,   email)   is   hidden   in   their   profile   until   booking   is   confirmed.   •   After   5   messages,   system   inserts   automated   message:   'You've   reached   the   pre-booking   limit.   Submit   a   booking   request   to   continue   chatting   with   this   owner.'     BYPASS   PREVENTION:   The   5-message   pre-booking   limit   and   phone/link   masking   are   non-negotiable   business   rules   that   protect   Roomzy's   commission   model   and   student   safety .   These   rules   cannot   be   configured   off   by   owners   or   students.   Only   Super   Admin   can   adjust   the   message   limit   via   platform   settings. Platform   Business   Rules     These   rules   govern   all   backend   module   behavior.   They   are   enforced   at   the   service   layer,   never   only   at   the   frontend.   Violation   of   any   rule   must   return   a   clear   error   response   with   a   specific   error   code.     11.1   Verification   Rules   RULE   V1:   An   owner   with   verification_status   !=   'verified'   cannot   submit   hostel   listings.   POST   /owner/hostels   returns   HTTP   403   with   error   code   OWNER_NOT_VERIFIED.     RULE   V2:   A   hostel   with   status   !=   'verified'   does   not   appear   in   public   search   results.   The   hostel   search   query   always   includes   WHERE   hostels.status   =   'verified'.   No   frontend   flag   can   override   this.     RULE   V3:   Only   verified   hostels   from   verified   owners   with   at   least   one   room   having   available_beds   >   0   are   shown   by   default   in   search.   This   triple-filter   is   the   foundation   of   platform   trust.     RULE   V4:   Rejected   owners   and   hostels   can   re-apply   by   resubmitting   documents   or   corrected   listing   details.   No   permanent   rejection   exists   in   Phase   1.     11.2   Booking   Rules   RULE   B1:   Booking   duration   must   be   exactly   1,   3,   6,   or   12   months.   Any   other   value   returns   HTTP   400   with   error   code   INVALID_BOOKING_DURA TION.     RULE   B2:   A   student   cannot   have   two   simultaneous   pending   or   approved   booking   requests   for   the   same   room.   Enforced   by   UNIQUE   constraint   +   service   layer   check.     RULE   B3:   A   room   with   available_beds   =   0   cannot   accept   new   booking   requests.   POST   /student/bookings   returns   HTTP   409   with   error   code   ROOM_NOT_A VAILABLE.     RULE   B4:   Student   has   exactly   24   hours   after   owner   approval   to   complete   payment.   A   Bull   queue   job   auto-cancels   and   returns   room   to   available   if   payment_deadline   passes. Security   Architecture     12.1   Authentication   Security   •   All   passwords   hashed   with   bcrypt   at   cost   factor   12   before   storage.   Password   is   never   returned   in   any   API   response.   •   JWT   tokens   signed   with   RS256   algorithm   using   private   key.   Public   key   used   for   verification   —   never   the   same   key.   •   Admin   JWT   tokens   expire   in   24   hours.   User   tokens   expire   in   7   days.   Refresh   token   (30   days)   extends   sessions.   •   Login   rate   limiting:   5   failed   attempts   per   IP   per   15   minutes   —   enforced   by   @nestjs/throttler.   •   Failed   login   errors   are   generic   ('Invalid   credentials')   —   never   reveal   whether   email   exists   in   system.   •   Admin   accounts   cannot   be   registered   via   public   /auth/register   —   provisioned   directly   by   Super   Admin.     12.2   API   and   Input   Security   •   All   request   bodies   validated   by   NestJS   ValidationPipe   with   class-validator   DTOs   —   invalid   inputs   return   400   with   field-level   error   details.   •   SQL   injection   prevention:   All   database   queries   use   TypeORM/Prisma   parameterized   queries.   Raw   SQL   never   used   with   user   input.   •   XSS   prevention:   All   user-submitted   text   is   sanitized   with   DOMPurify   equivalent   before   storage.   •   CORS   configured   with   whitelist:   only   production   frontend   domain   and   localhost   (development)   allowed.   •   All   production   API   traffic   over   HTTPS   only.   HTTP   requests   are   redirected   to   HTTPS.   •   File   upload   validation:   type,   size,   and   MIME   type   checked   before   Cloudinary   upload.   Max   10MB,   allowed   types:   JPEG/PNG/WebP/PDF.     12.3   Business   Logic   Security   •   Resource   ownership   guards   on   all   CRUD   operations   —   an   owner   cannot   edit   another   owner's   hostel   even   with   a   valid   token.   •   Booking   state   transitions   validated   at   service   layer   —   invalid   transitions   (e.g.,   pending   →   completed)   return   409   Conflict.   •   Commission   rate   is   read   from   database   at   payment   time   —   any   frontend-submitted   commission   rate   is   ignored   entirely.   •   Review   gating   enforced   at   service   layer   —   even   if   a   modified   API   call   is   made,   the   service   validates   booking   ownership   and   completion   status.
